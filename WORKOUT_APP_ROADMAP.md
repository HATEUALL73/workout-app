# WORKOUT_APP_ROADMAP.md

## 1. Назначение документа

Этот файл — единая утверждённая спецификация развития приложения `workout-app`.

Codex должен использовать его как основной источник требований по следующим функциям. Нельзя внедрять все пункты одним большим изменением. Работа выполняется последовательно, небольшими контролируемыми этапами.

---

## 2. Правила работы Codex

Для каждого пункта:

1. Сначала изучить текущий код и существующую архитектуру.
2. Сообщить:
   - какие файлы потребуется изменить;
   - нужна ли миграция базы данных;
   - какие зависимости есть от других пунктов;
   - какие риски регрессии обнаружены.
3. До подтверждения пользователя код не менять.
4. После подтверждения реализовать только текущий пункт.
5. Не выполнять попутный рефакторинг, не относящийся к задаче.
6. Запустить проверки:
   - `npx.cmd tsc --noEmit`
   - `npx.cmd expo install --check`
   - `git diff --check`
   - дополнительные тесты, если они есть в проекте.
7. Показать список изменённых файлов и кратко описать изменения.
8. Подготовить ручной сценарий проверки в Expo Go или development build.
9. После успешной проверки создать отдельный коммит.
10. Не переходить к следующему пункту без подтверждения пользователя.

Один пункт — один логический набор изменений и отдельный коммит. Крупный пункт можно разбить на несколько последовательных коммитов, если это снижает риск.

## 3. Текущее состояние проекта

Проект: `workout-app`

Ориентировочный стек:

- Expo SDK 56
- React Native 0.85.3
- React 19.2.3
- TypeScript 6, strict mode
- Expo Router
- Expo SQLite
- Expo Audio
- Expo Haptics
- React Native SVG
- EAS Build

Уже реализовано и не должно быть сломано:

- карточки упражнений;
- выполнение упражнения по подходам;
- автоматический запуск отдыха после промежуточных подходов;
- после последнего подхода упражнение завершается без запуска отдыха;
- одна запись в истории после завершения упражнения;
- автоподстановка прошлых веса и повторов;
- ручной ввод имеет приоритет над автоподстановкой;
- таймер отдыха;
- безопасная работа в Expo Go без падения из-за `expo-notifications`;
- предупреждения о рисках в текущем упрощённом виде.

Фоновые уведомления и гарантированная работа уведомлений на заблокированном экране пока отложены до development/EAS build.

## 4. Общие ограничения

Нельзя без необходимости:

- удалять или сбрасывать существующую историю;
- менять seed и упражнения вне текущей задачи;
- ломать текущую логику таймера;
- создавать несколько исторических записей за одно завершённое упражнение;
- смешивать результаты разных упражнений;
- копировать рабочий вес между разными упражнениями;
- автоматически менять постоянный план тренировки при временной замене;
- блокировать выполнение упражнения только из-за предупреждения о риске;
- хранить новые сложные данные в одной текстовой строке;
- делать массовый рефакторинг всего проекта.

Приоритет данных:

1. Черновик текущей тренировки.
2. Завершённая запись за сегодня.
3. Последняя завершённая тренировка для автоподстановки.
4. Значения по умолчанию.

# ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

## Пункт 1. Автоподстановка прошлого веса и повторов

**Статус: реализовано.**

Текущее поведение нужно сохранить:

- при отсутствии черновика подставляются значения последней тренировки;
- ручное изменение не затирается;
- завершённое сегодня упражнение повторно не автозаполняется;
- после внедрения пункта 6 значения каждого подхода подставляются отдельно;
- при увеличении количества подходов дополнительный подход копирует последний подход прошлой тренировки;
- при уменьшении количества подходов лишние прошлые значения не используются.

Регрессионная проверка обязательна после изменений структуры истории.

## Пункт 2. Автоматический выбор тренировочного дня

### Логика

- понедельник и вторник → программа понедельника;
- среда и четверг → программа среды;
- пятница, суббота и воскресенье → программа пятницы.

### Требования

- использовать локальную дату устройства;
- автоматический выбор выполняется при первичной инициализации экрана;
- существующее ручное переключение дней сохраняется;
- после ручного переключения приложение не должно самопроизвольно возвращать вычисленный день;
- на первом этапе не выполнять автоматическую смену дня при возвращении приложения из фона;
- автоматическая смена после полуночи может быть добавлена позже вместе с надёжным сохранением черновика;
- БД, историю, таймер, уведомления и логику подходов не менять.

### Приёмка

- во вторник открывается понедельник;
- в четверг открывается среда;
- в воскресенье открывается пятница;
- ручное переключение остаётся выбранным до следующей инициализации;
- введённые данные не теряются.

## Пункт 3. История по конкретному упражнению

### Интерфейс

Из карточки упражнения доступна кнопка:

`История упражнения`

Она открывает отдельный экран упражнения.

### Содержимое

В верхней части:

- название упражнения;
- дата последней тренировки;
- последний результат;
- лучший результат.

Ниже — тренировки от новых к старым.

Для каждой тренировки:

- дата;
- все выполненные подходы;
- вес и повторы;
- пометка о временной замене;
- причина замены, если она была.

### Пагинация

- при первом открытии показывать последние 10 тренировок;
- внизу кнопка `Показать ещё`;
- каждая загрузка добавляет следующую порцию записей;
- всю историю сразу не загружать.

### Старые данные

Старые записи, где сохранён один результат, не удалять. Показывать их как один сохранённый подход с пометкой:

`Детальные данные по подходам не сохранялись`.

### Не входит в этот пункт

- графики;
- редактирование;
- удаление;
- рекомендации по увеличению веса.

---

## Пункт 4. Редактирование и удаление записи за сегодня

### Доступность

Редактировать и удалять можно только запись за текущий календарный день.

Действия доступны:

- в карточке выполненного упражнения;
- у сегодняшней записи на экране истории упражнения.

### Редактирование

- открыть все сохранённые сегодня подходы;
- разрешить изменить вес и повторы каждого подхода;
- обновить существующую запись;
- не создавать дополнительную запись;
- дату и упражнение не менять.

### Удаление

Перед удалением обязательно показать подтверждение:

`Удалить сегодняшнюю запись?`

После подтверждения:

- удалить только сегодняшнюю запись этого упражнения;
- вернуть упражнение в незавершённое состояние;
- сохранить введённые вес и повторы в карточке как текущий черновик;
- позволить исправить значения и повторно завершить упражнение;
- не затрагивать прошлые тренировки и другие упражнения;
- не запускать таймер и уведомления.

---

## Пункт 5. Сохранение черновика тренировки после перезапуска

### Что сохранять

- выбранный тренировочный день;
- введённый вес и повторы каждого подхода;
- количество завершённых подходов;
- статус незавершённых упражнений;
- временные замены упражнений;
- данные, возвращённые в черновик после удаления сегодняшней записи;
- время обновления;
- дату тренировки.

### Когда сохранять

Автоматически после:

- изменения веса;
- изменения повторов;
- завершения промежуточного подхода;
- ручного выбора дня;
- выбора или отмены временной замены;
- удаления сегодняшней записи с возвратом данных в черновик.

Можно применять небольшую задержку записи, чтобы не обращаться к БД после каждого символа.

### Восстановление

Приоритет:

1. Черновик.
2. Завершённая запись за сегодня.
3. Последняя завершённая тренировка для автоподстановки.

После завершения упражнения:

- создать историческую запись;
- удалить черновик этого упражнения;
- не хранить одновременно завершённую запись и активный черновик одного упражнения.

### Черновик прошлого дня

При обнаружении старого черновика показать:

`Найдена незавершённая тренировка за <дата>.`

Действия:

- `Продолжить тренировку`;
- `Начать новую`.

При продолжении восстановить исходный день, подходы и прогресс.

При начале новой не удалять старый черновик молча. Сначала запросить подтверждение удаления. Архивирование можно добавить позже.

### Не сохранять

- состояние открытой карточки;
- позицию прокрутки;
- открытый экран;
- состояние таймера отдыха;
- запланированное уведомление.

---

## Пункт 6. Хранение всех подходов

### Модель

Одно завершённое упражнение — одна запись тренировки, внутри которой хранится список всех выполненных подходов.

Для каждого подхода:

- порядковый номер;
- вес, если применимо;
- количество повторов;
- признак завершения;
- время завершения.

Подходы лучше хранить отдельными строками БД, связанными с записью упражнения.

### Сохранение

- промежуточные подходы остаются в черновике;
- после последнего подхода создаётся одна запись упражнения со всеми подходами;
- черновик упражнения удаляется.

### Лучший результат

В компактной карточке показывать вычисляемый лучший подход:

1. наибольший вес;
2. при одинаковом весе — больше повторов;
3. без веса — максимум повторов.

Не хранить лучший подход как независимое дублирующее поле без необходимости.

### Автоподстановка

- переносить прошлые подходы по порядку;
- если новых подходов больше, дополнительные копируют последний прошлый подход;
- если новых подходов меньше, лишние прошлые подходы игнорируются.

### Старые записи

Старую запись с одним результатом преобразовать логически в запись с одним подходом. Историю не удалять.

---

## Пункт 7. Категории риска

### Категории

- локоть;
- колено;
- спина.

Одно упражнение может иметь несколько рисков.

### Данные риска

Для каждого риска хранить:

- область;
- уровень;
- краткую причину;
- рекомендацию по осторожному выполнению, если она есть.

Уровни:

- `Осторожно`;
- `Не рекомендуется`.

Упражнение не блокируется принудительно.

### Настройки

В настройках отдельные переключатели:

- Локоть — вкл./выкл.
- Колено — вкл./выкл.
- Спина — вкл./выкл.

Риски остаются привязанными к упражнениям, но предупреждения показываются только для включённых ограничений.

### Отображение

В свёрнутой карточке — компактные метки зон.

В раскрытой карточке — блок с объяснением причин.

Если рисков несколько, визуальный стиль определяется самым серьёзным активным уровнем.

---

## Пункт 8. Альтернативы для рискованных упражнений

### Назначение

При активном риске пользователь может открыть список более безопасных альтернатив.

### Подбор

Учитывать:

- основную мышечную группу;
- тип движения;
- оборудование;
- все включённые ограничения;
- уровень риска альтернативы;
- степень соответствия исходному упражнению.

Категории:

- лучшее соответствие;
- хорошая замена;
- частичная замена.

Если полностью безопасной альтернативы нет, сообщить об этом прямо и показать варианты, которые снижают только часть рисков.

### Выбор

После выбора:

- замена действует только на текущую тренировку;
- постоянный план не меняется;
- отображается исходное упражнение и причина замены;
- в историю записывается фактически выполненное упражнение;
- сохраняется связь с исходным упражнением;
- результаты не смешиваются в графиках.

### Вес и повторы

- вес исходного упражнения не переносить;
- использовать последнюю историю выбранной альтернативы;
- при отсутствии истории оставить поля пустыми.

### Возврат

До завершения альтернативы доступна кнопка `Вернуть исходное упражнение`.

Постоянную замену пока не добавлять.

### Количество подходов

Для унификации механизма временных замен рекомендуется сохранять количество подходов исходного упражнения, а вес и повторы подставлять из истории альтернативы. Перед реализацией Codex должен отдельно подтвердить это правило с пользователем, поскольку оно не было окончательно утверждено в обсуждении.

## Пункт 9. Быстрые кнопки веса и повторов

### Интерфейс

Все кнопки постоянно видны под каждым подходом.

Вес:

`−5  −2,5  [значение]  +2,5  +5`

Повторы:

`−2  −1  [значение]  +1  +2`

Конкретные шаги веса зависят от упражнения.

### Требования

- изменение относится только к выбранному подходу;
- ручной ввод через клавиатуру сохраняется;
- вес и повторы не могут быть ниже нуля;
- дробные значения отображаются без ошибок плавающей точки;
- лишние нули не показываются;
- любое изменение считается ручным вводом и имеет приоритет над автоподстановкой;
- изменения сразу сохраняются в черновик.

### Копирование подхода

Для второго и следующих подходов доступна кнопка:

`Повторить предыдущий`

Она копирует вес и повторы предыдущего подхода.

### Шаг веса

Приоритет:

1. индивидуальный шаг упражнения;
2. общий шаг из настроек.

## Пункт 10. Экран текущей тренировки

### Запуск

Отдельная кнопка:

`Начать тренировку`

До нажатия время не считается.

После старта длительность рассчитывается по отметкам времени, а не только по активному секундному таймеру.

### Верхняя часть

Показывать:

- тренировочный день;
- дату;
- количество выполненных упражнений;
- количество оставшихся упражнений;
- длительность тренировки.

### Список упражнений

Статусы:

- не начато;
- начато;
- текущее;
- завершено;
- временно заменено.

Пользователь может вручную открыть любое упражнение. Прогресс других упражнений не теряется.

### Текущее упражнение

Показывать:

- название;
- подходы;
- отдых;
- поля веса и повторов;
- быстрые кнопки;
- завершение подхода;
- ручной запуск отдыха;
- риски;
- кнопку временной замены.

### После завершения упражнения

Не переходить автоматически.

Остаться на текущем экране и показать кнопку:

`Перейти к следующему упражнению`

Пользователь сам выбирает момент перехода.

### Завершение тренировки

После выполнения всех упражнений показать итог:

- упражнения;
- подходы;
- длительность.

Досрочное завершение допускается с предупреждением. Незавершённые данные сохраняются, чтобы тренировку можно было продолжить позже.

## Пункт 11. Графики прогресса

### Расположение

Графики находятся на экране истории конкретного упражнения.

Переключатель:

`История | Графики`

### Показатели

Отдельные графики:

1. лучший рабочий вес;
2. максимальное количество повторов;
3. общий объём.

Объём:

`сумма(вес × повторы)` по всем завершённым подходам.

Для упражнений без веса график объёма в килограммах не показывать.

### Периоды

- 1 месяц;
- 3 месяца;
- 6 месяцев;
- всё время.

По умолчанию — 3 месяца.

### Поведение выбора

- при первом открытии показывать лучший рабочий вес;
- затем запоминать последний выбранный показатель;
- желательно запоминать и последний выбранный период.

### Точки графика

При нажатии показывать:

- дату;
- лучший вес;
- лучший подход;
- общий объём;
- кнопку открытия подробной тренировки.

### Старые данные

Старые записи с одним подходом участвуют в графике, но для объёма помечаются как неполные. Нельзя искусственно умножать старый результат на предполагаемое количество подходов.

### Замены

Результаты разных упражнений не смешивать. Замена попадает в график фактически выполненного упражнения.

---

## Пункт 12. Замена упражнения, если тренажёр занят

### Назначение

Пользователь может заменить упражнение на похожее, если нужный тренажёр занят долгое время.

Причина замены:

`Тренажёр занят`

### Подбор

Учитывать:

1. основную мышечную группу;
2. тип движения;
3. роль в тренировке;
4. другое оборудование;
5. активные ограничения пользователя.

Категории соответствия:

- лучшее;
- хорошее;
- частичное.

### Ограничение

Замена разрешена только до завершения первого подхода.

После первого завершённого подхода:

- кнопку замены заблокировать;
- позволить перейти к другому упражнению программы;
- позже пользователь может вернуться к начатому упражнению.

### После выбора

- постоянный план не меняется;
- замена действует только на текущую тренировку;
- история хранит фактически выполненное упражнение;
- сохраняются исходное упражнение и причина замены;
- результаты разных упражнений не смешиваются.

### Вес и повторы

- данные исходного упражнения не копировать;
- использовать историю альтернативы;
- при отсутствии истории оставить поля пустыми.

### Возврат

До завершения первого подхода можно вернуть исходное упражнение.

Черновики исходного упражнения и временной альтернативы не должны теряться при переключении между ними.

### Отличие от обычного перехода

Открытие другого упражнения из текущего плана не считается заменой. Это обычное переключение активного упражнения.

# 6. Общая модель временной замены

Пункты 8 и 12 должны использовать один технический механизм.

Запись временной замены должна содержать:

- исходное упражнение;
- фактически выполненное упражнение;
- причину:
  - риск;
  - тренажёр занят;
- дату;
- тренировочный день;
- количество подходов;
- статус;
- связь с исторической записью.

Причины и правила фильтрации различаются, но модель данных и основной интерфейс должны быть общими.
